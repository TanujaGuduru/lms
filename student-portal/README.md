# CodeGurukul Student Portal

Plain PHP + HTML/CSS/JavaScript — same architectural style as the existing
Admin panel (`../app`, `../bootstrap`, manual MVC, no Composer dependencies at
runtime). Built specifically to deploy on **GoDaddy shared hosting with no
manual installation steps** — no Composer, no npm, no build tooling required
on the server. Everything here is just files you upload.

Full design background: `../docs/student-module/` (35-document architecture
series — schema, workflows, APIs, AI design, scaling, infra). **One load-bearing
correction to that series**: it was originally written assuming AWS + Agora +
Pinecone etc. as the target, with a GoDaddy-compatible fallback path. The
actual decision, made explicit partway through implementation, is the
opposite — **this runs on GoDaddy shared hosting only, for the long term, with
no external cloud service of any kind** (not Agora, not Pusher/Ably, not S3 —
the only narrow exception is a direct, pay-per-call AI provider API for
features that are physically impossible without an LLM somewhere). Every
place this matters is called out explicitly below and in the relevant
controller's docblock.

## No-cloud architecture decisions

| Originally designed as | Actually built as | Why |
|---|---|---|
| Agora.io (managed RTC) | **Pure peer-to-peer WebRTC**, full mesh, signaled via this app's own REST API (polled, since shared hosting can't hold a WebSocket open) | No external account, no per-minute billing, no server this hosting model could run anyway. Real, accepted tradeoff: works well for the small class sizes this platform actually uses; doesn't scale to a large lecture without a relay server shared hosting can't provide. No TURN relay either — the free public Google STUN server resolves NAT for most networks, but a peer behind a symmetric/restrictive NAT can still fail to connect. |
| AWS S3 + signed URLs | **Local disk storage** (`storage/app/`) + a signed-token download endpoint this app generates and verifies itself | Zero external account. The signing logic (HMAC + expiry) is real, not a stub — it just signs a local path instead of an S3 object key. |
| AWS MediaConvert (adaptive transcoding) | **A single progressively-downloaded file per recording** | Shared hosting can't run a transcoding pipeline. An honest simplification, not a placeholder. |
| Pinecone / pgvector (RAG) | **Dropped — no retrieval at all.** The AI tutor's system prompt names the linked lesson by title; it answers from the model's own general knowledge, not retrieved course content. | No cloud vector DB of any kind. Less grounded than the original design, accepted as the cost of zero cloud infrastructure rather than hidden. |
| SSE streaming for AI replies | **Dropped — single blocking request/response.** | Shared hosting's mod_php/FastCGI output buffering isn't reliable enough to promise a real token-by-token stream. A real latency tradeoff (full wait, not progressive display), not a missing feature. |
| Direct Anthropic/OpenAI API calls | **Built — `App\Core\AiGateway`, direct cURL, no SDK** — kept as the one accepted external dependency | No way to run a capable LLM on shared PHP hosting; this is a plain API-key HTTPS call, not infrastructure to manage. |
| Pinecone (originality/plagiarism check) | **An AI-judged similarity rating** via the same `AiGateway`, run by a cPanel cron job | Cruder than real embedding-similarity, but zero additional cloud account — reuses the one already-accepted exception instead of adding a second one. |
| S3-hosted PDF (`parent_reports.pdf_url`) | **`App\Core\SimplePdf`** — a hand-rolled, single-font, no-library PDF writer, generated by a cPanel cron job, served via the same local signed-URL mechanism as Materials/Recordings | No S3, no PDF-generation service, no Composer dependency — consistent with everything else in this table. |
| "Nightly batch job" infra (snapshots, reports, originality checks) | **GoDaddy cPanel's own Cron Jobs feature** | Ordinary shared-hosting functionality, not a cloud service — see `DEPLOYMENT.md` for the exact cron lines. |
| HTML+CSS certificate template rendering (`certificate_templates.html_template`/`css_styles`) | **Dropped — `App\Core\SimplePdf` instead**, same writer as parent reports, now with basic centered text | An HTML+CSS-to-PDF engine needs a headless browser or a heavy library — neither available without Composer or a cloud service. The template system stays unused; the certificate is plain but real. |
| Pusher/Ably (Collaborative Coding's real-time Yjs sync) | **Reuses `ClassroomController`'s pure-P2P-WebRTC + polling-signal pattern** (`collab_signals`, a parallel table to `webrtc_signals`) | Same reasoning as Live Classroom — no relay server this hosting model can provide. No client-side collaborative editor exists yet; this is backend-only, by deliberate scope boundary (see 04h). |
| Speech-to-text (`POST /notes/voice-transcribe`) | **Not implemented** | Anthropic's API has no audio-transcription capability; a real implementation needs a second external provider (e.g. OpenAI Whisper) for this one feature alone — deferred rather than adding a second accepted cloud dependency silently. |
| Direct-to-S3 keystroke event streams (Code Replay) | **Not implemented — the whole feature, not just the storage layer** | `code_replay_sessions.workspace_id` is a required FK to `code_workspaces`, which nothing in this build ever populates (no Coding Sandbox at all). There's no live coding session to replay, regardless of where the bytes would be stored. |
| AES-128 HLS encryption + per-validation `decryption_key` (Offline Access) | **Plain signed local-file URLs** (`FileStorage`, longer expiry) — no encryption-at-rest at all | A real DRM pipeline needs infrastructure (segmenting + encryption) this build never has even for online playback. The access-control shape (token, expiry, revoke, re-validation) survives; only the cryptographic payload is gone. |
| Google/Outlook OAuth2 + Apple CalDAV (Calendar `connect`/`reconnect`) | **Not implemented** — `providers`/`disconnect` are real, reading/writing actual `calendar_connections` rows | Needs real developer credentials from 3 separate providers, a distinct undertaking from the AI Gateway's single API key. Deferred rather than half-built with placeholder client IDs that couldn't authenticate anyone. |
| AWS Bedrock transport + a config flag to switch hosting environments (05a §1) | **N/A — there is only ever one hosting target** | 05a's "hosting duality" exists for a deployment that might run on AWS *or* GoDaddy. This build is GoDaddy-only for the long term by explicit, standing decision — the direct-HTTPS-API path `AiGateway` already used is the only path that will ever exist here, not one branch of a config flag. |

## Layout

```
student-portal/
├── api/            PHP backend — deploys to public_html/api/
│   ├── app/Core/           Database, Auth, Router, Request, Controller, Logger,
│   │                       FileStorage (signed local-file URLs — no S3)
│   ├── app/Controllers/Api Auth, Account, ParentConsent, Wallet, Placement, Batch,
│   │                       Schedule, Classroom (WebRTC signaling + chat),
│   │                       Attendance, Recording, Lesson, Material, File, Health
│   ├── app/Middleware/     AuthMiddleware (Bearer-token gate)
│   ├── bootstrap/app.php   env loading, error handlers, security headers, router
│   ├── config/             app.php (incl. APP_KEY), database.php
│   ├── database/schema_student_portal.sql   the full additive schema
│   ├── storage/app/        local file storage (materials/, recordings/) — never web-reachable directly
│   ├── routes/api.php
│   ├── index.php           entry point
│   └── .htaccess           blocks app/config/storage/etc., routes everything else to index.php
└── web/             Static frontend — deploys to public_html/ (domain root)
    ├── index.html, login.html, dashboard.html, classroom.html (live video call UI)
    ├── assets/css/  style.css, classroom.css
    ├── assets/js/   api.js (fetch client), login.js, dashboard.js, classroom.js (WebRTC mesh)
    └── .htaccess
```

This mirrors the **same domain, different paths** layout chosen for this
project: `yourdomain.com/` serves the frontend, `yourdomain.com/api/` serves
the backend, both from one GoDaddy hosting account.

## Why this shape on GoDaddy specifically

- GoDaddy shared hosting's primary-domain document root is fixed at
  `public_html/`. There's no per-path document-root override, so the API
  can't live in an isolated webroot the way it could on a dedicated subdomain —
  it has to be a real, web-reachable subfolder.
- `api/index.php` plus `api/.htaccess` therefore plays exactly the role
  `public/index.php` + `public/.htaccess` plays in the Admin panel: the
  `.htaccess` rule blocking `app/`, `bootstrap/`, `config/`, `routes/`,
  `storage/`, `database/` is the only thing keeping those folders off the
  public web, since there's no separate non-web-accessible directory to put
  them in.
- The frontend is plain static HTML/CSS/JS specifically because GoDaddy
  shared hosting has no Node.js runtime — there is nothing to "run" for the
  frontend, only files to upload. No `npm install`, no build step, ever,
  on the server.

## Deploying to GoDaddy

1. **Database**: the Admin panel's `database/schema.sql` must already be
   imported. Then import `api/database/schema_student_portal.sql` (via
   phpMyAdmin, or `mysql codegurukul < schema_student_portal.sql` if you have
   shell access) — it only adds/extends tables, never touches existing data.
2. **Backend**: upload the entire contents of `api/` to `public_html/api/`
   on the server (FTP, cPanel File Manager, or git). Copy `.env.example` to
   `.env` inside `public_html/api/` and fill in the real `DB_*` credentials
   (same database the Admin panel already uses) and `APP_URL`/`FRONTEND_URL`.
3. **Frontend**: upload the entire contents of `web/` to `public_html/`
   (the domain root).
4. Visit `https://yourdomain.com/api/v1/health` — should return
   `{"success":true,"data":{"status":"ok","database":true}}`. If `database`
   is `false`, double-check `public_html/api/.env`.
5. Visit `https://yourdomain.com/` — should redirect to the login page and
   let an existing `users` row (role Student or Parent, `status='active'`)
   sign in through to the dashboard placeholder.

No `composer install`, no `npm install`, no build step — both folders are
deployed exactly as they are in this repository.

## What's actually implemented (this pass) — verified, not just reviewed

This covers all of `docs/student-module/04a` (Conventions, Auth/Account,
Enrollment, Credit Wallet) — not yet the full 39-lifecycle-phase build-out.
Everything below was actually installed and run locally (PHP 8.3 + an
isolated throwaway MySQL 8.4 instance) and exercised via `curl`, not just
read through. Real bugs only showed up under actual execution:

- **Full additive database schema** (`api/database/schema_student_portal.sql`)
  — every table/column change across all 35 architecture documents, with
  foreign-key column types corrected against the real existing schema (most
  existing tables use `INT UNSIGNED` ids, not `BIGINT`). Confirmed by
  importing it against the Admin panel's real `schema.sql` — 56 existing
  tables + 67 new ones (66 from the architecture docs + `parent_consent_otps`,
  a gap-fill described below), zero errors.
- **Auth**: login, logout (revokes token), refresh (rotates token), `/auth/me`,
  forgot-password, reset-password — all reusing the Admin panel's existing
  `api_tokens` table/mechanism, not a new auth system. `linked_students` (with
  per-link visibility booleans) is included in both `/auth/login` and
  `/auth/me` responses when the caller is a parent, per 04a's documented shape.
- **Account/Profile**: `GET`/`PATCH /account/profile`, `POST
  /account/complete-onboarding` — `interests`/`goals` validated against a
  small controlled vocabulary (not free text), exactly as 04a specifies.
- **Parent Consent**: `GET /parent/consent-requests`, `POST
  .../initiate|grant|revoke` — real OTP issuance/verification, and granting a
  student's last pending consent gate flips their account `pending → active`
  in the same DB transaction, per 04a's explicit requirement.
- **Credit Wallet**: `GET /wallet`, `GET /wallet/transactions` (paginated),
  `POST /wallet/freeze` (scheduled, not immediate — status stays `active`
  until a future cron job would flip it), `POST /wallet/resume` (reverses an
  actual freeze with expiry-date extension, or cancels a still-pending one),
  `POST /wallet/purchase` (creates a real `payments` row; the actual gateway
  redirect is stubbed since no Razorpay/Stripe credentials exist yet). Parent
  access requires `?student_id=` and is gated by that link's
  `can_view_billing`.
- A `health` endpoint (`/api/v1/health`) that checks DB connectivity.

Bugs caught only by actually running this, not by review (8 total across both
passes):
1. `COMMENT` was placed after `AFTER` in one `ALTER TABLE ADD COLUMN` — MySQL
   requires the opposite order; a real syntax error.
2. Dropping `attendance`'s old unique key failed because it was the only index
   covering `batch_id`'s foreign key — needed a replacement index added first.
3. `routes/api.php` registered routes under prefix `v1`, but the real request
   path (both on GoDaddy and via PHP's dev server) is always `/api/v1/...` —
   every route 404'd until the prefix became `api/v1`.
4. `/auth/me` was leaking `password_reset_token`/`email_verification_token`
   in the response — only `password_hash` and friends were being stripped.
5. `parent_student_links` has a composite primary key `(parent_id,
   student_id)` — no surrogate id exists for 04a's documented `{linkId}` path
   parameter to mean. Resolved by treating it as `student_id`, since the
   parent half of the link is always the authenticated caller.
6. No table anywhere in the 35-document series actually stored the OTP that
   `POST /parent/consent/{id}/grant` validates against — added
   `parent_consent_otps` (same "discover the gap, fill it, say so" pattern as
   the `assignments`/`live_quizzes` gaps caught earlier in this project).
7. The Router had no `patch()` method (only get/post/put/delete existed) —
   `PATCH /account/profile` had nowhere to register until it was added.
8. `WalletController::resume()` could be called twice in a row and both calls
   would "succeed" — the pending-freeze lookup found the same old `frozen` log
   row even after a `resumed` row had already cancelled it, because it didn't
   check whether that `frozen` row was still the *latest* entry.

Verified end to end via `curl` for every endpoint above, including the
negative paths: wrong password vs. unknown email return the identical generic
401 (no account-enumeration leak), an authenticated call after logout/refresh
correctly 401s on the old token, a password reset revokes every existing
token, wrong/expired OTP codes are rejected before the consent grant, double
`resume` calls and double `freeze` calls are both correctly rejected.

## Phase 4b — Placement, Batch, Scheduling (also implemented + verified)

`docs/student-module/04b-apis-assessment-scheduling.md` is fully wired up too:

- **Placement**: `status`/`start`/`answer`/`submit`/`result`/`request-recheck`
  — real attempt lifecycle against `exam_attempts`/`exam_responses`, with
  deterministic per-attempt question shuffling (`mt_srand($attemptId)`),
  objective auto-grading, and the two-shape result response (unreviewed AI
  guess vs. mentor-confirmed) the doc specifies. Communication-section AI
  scoring is left `NULL` pending a real AI Gateway — not invented here.
- **Batch**: `current` (teacher, schedule, batchmate *count* only — never
  other students' identities) and `waitlist-status` (a coarse wait-time
  bucket, not a false-precision queue position).
- **Schedule**: `upcoming`/`calendar`/`classes/{id}` (server-computed
  timezone conversion via PHP's `DateTime`, never client-side UTC math) and
  full `reschedule-requests`/`teacher-change-requests` CRUD with the monthly
  reschedule limit computed live (never a stale counter) and an
  insufficient-notice window check.

Two real bugs found only by running it:
1. `requestRecheck()` invalidates the old attempt, but `start()`'s one-shot
   guard only excluded `abandoned` status, not `invalidated` — a recheck
   that was supposed to unblock a new attempt didn't.
2. `status()` had the same gap from the other direction: after a recheck, it
   kept reporting the stale invalidated attempt's old mentor-confirmed result
   as `done` instead of `required`.

One unresolvable-by-spec gap, resolved with the simplest defensible
interpretation rather than guessing a specific business rule: nothing in the
35-document series says which exam *is* "the" placement test for a course, or
which courses require one at all. Resolved as `exams.type='placement'` scoped
to (or falling back from) the course, with every course currently treated as
requiring one.

## Phase 4c — Live Classroom, Attendance, Recordings, Video Library, Materials

`docs/student-module/04c-apis-classroom-content.md` is wired up, **rebuilt
around the no-cloud decision above** rather than as originally drafted:

- **Live Classroom — pure peer-to-peer WebRTC** (`ClassroomController`):
  `join` (the same three real authorization checks the original "security-
  critical endpoint" design called for — live `batch_students` membership,
  account status, class-open/join-window — now returning ICE server config
  and the current participant list instead of an Agora token),
  `heartbeat`/`leave` (attendance tracking *and* presence — "who's in the
  room" is derived from recent heartbeats, no separate presence table),
  `signal`/`signal` (poll) — the entire WebRTC signaling channel (SDP
  offers/answers, ICE candidates), addressed pairwise per the mesh topology,
  polled over plain HTTP every ~1.5s since shared hosting can't hold a
  WebSocket open. `chat`/`chat` (poll) — a lightweight room-wide text chat,
  same polling pattern. A real, working video-call UI exists at
  `web/classroom.html` (video grid, mic/camera toggle, chat panel) — this
  isn't only an API, it's a page you can actually open in two browser tabs
  and have a call. Heartbeat/leave still use a deliberately simplified
  single join/leave window per session rather than `03c`'s full
  multi-interval reconnect-gap union — a documented scope cut, not an
  oversight.
- **Attendance**: `history`/`summary` — read-only, exactly as specified (no
  POST/PATCH exists at all; overrides are Admin/Teacher-portal-only).
- **Recordings**: `index`/`show`/`progress`/`bookmarks`/`save-to-note`/
  `search` — access control is a live `batch_students` join against the
  student's full enrollment *history* (not just current batches), 404 (never
  403) on anything outside it. `search` uses a real MySQL `FULLTEXT` index
  on `transcript_text` — added in this pass, since `01b`'s stated v1 search
  strategy (FULLTEXT over a separate OpenSearch cluster) had never actually
  been created anywhere in the schema until a search endpoint needed it.
  Recorded video is a single file on local disk, served through the same
  signed-URL mechanism as Materials below — no adaptive-bitrate transcoding
  (shared hosting can't run that pipeline; an honest simplification).
- **Video Lecture Library**: `courses/{id}/modules` (with per-lesson
  progress joined in), lesson detail/progress (completion is a side effect
  of crossing the ~90% threshold, never a separate "mark complete" call)/
  bookmarks.
- **Materials**: course materials list, `download` (logs the access event
  unconditionally, returns a signed local-file URL via `App\Core\FileStorage`),
  `versions`. The signing is real (HMAC-SHA256 over the path + expiry, using
  this app's own `APP_KEY`) — there's no S3 behind it at all, by design, not
  as a stand-in for one.

Bugs found only by running this (on top of the 8 from earlier phases):
1. `mime_content_type()` and `mb_strlen()` both crashed with "undefined
   function" — the `fileinfo` and `mbstring` PHP extensions weren't enabled
   in this machine's fresh `php.ini` (PHP ships both as extensions, not
   built-ins). Fixed locally by enabling them; **GoDaddy and virtually every
   shared host ship both enabled by default** since most CMSs require them,
   so this isn't expected to be a real deployment risk — noted here so it's
   not a silent assumption either.
2. A signed local-file token built from a deliberately malicious
   (`../../`-style) path was tested directly against `FileController::serve()`
   — correctly rejected by the `realpath()` containment check in
   `FileStorage::validate()` even though the HMAC signature itself was
   genuinely valid for that path, confirming the path-traversal defense is a
   real, independent second layer and not just a comment claiming one exists.

## Phase 4d — Assignments, AI Doubt Solver, AI Coding Assistant (no Coding Sandbox)

`docs/student-module/04d-apis-assignments-sandbox-ai.md` covers three
features; only two are built, by deliberate decision:

- **Coding Sandbox is not implemented at all — no `/workspaces/*` routes,
  no `code_workspaces` controller, nothing.** This platform never executes
  student-submitted code, full stop ("no coding live should be there" was
  an explicit, confirmed decision, not a GoDaddy workaround to find). The
  `code_workspaces`/`workspace_files`/`code_executions` tables from the
  original schema design are still present in
  `schema_student_portal.sql` (already-tested, harmless to leave) but
  nothing in this app ever reads or writes them — they're inert.
- **Assignments** (`AssignmentController`): course list, detail (with a
  request-time-computed `is_overdue`, never a stored column), own
  submission, draft autosave (`PUT .../submission`), and finalize
  (`POST .../submission/submit`). The one place Coding Sandbox's absence
  actually changes this doc's design: `type='code'` submissions were
  originally a `{workspace_id}` reference the server zips server-side —
  with no workspace at all, a code submission is just pasted code in
  `submission_text`, exactly like a text assignment. Still reviewable and
  gradeable by a teacher, just never run by anything.
- **AI Doubt Solver / AI Coding Assistant** (`AiController` +
  `App\Core\AiGateway`): real conversations with a live LLM (Anthropic's
  API, called directly over cURL — see `AiGateway`'s docblock for the
  full no-cloud reasoning), with five teaching-mode system prompts
  (`hint`/`explain`/`practice` for Doubt Solver, `debug`/`review` for
  Coding Assistant) that withhold the direct answer except via the
  explicit `POST .../escalate` "show me the fix" endpoint. Quota is
  enforced with an atomic check-and-reserve `UPDATE ... WHERE messages_used
  < quota_limit_messages` (the affected-row-count is the actual gate, not a
  separate read-then-write that a burst of concurrent requests could race),
  returning `429` with `resets_at` once exhausted. Near-duplicate questions
  (3+ similar asks in one conversation) silently flag the conversation for
  mentor review via a local `similar_text()` heuristic — the student's
  response is completely unaffected either way.
  Two things the original design called for that aren't here, both because
  they needed a cloud service this build doesn't have:
  - **No streaming (SSE).** Shared hosting's mod_php/FastCGI output
    buffering isn't reliable enough to promise a real token-by-token
    stream — this is a single blocking request/response instead. A real
    latency tradeoff (the student waits for the full reply rather than
    seeing it appear word-by-word), accepted rather than hidden.
  - **No RAG over course content.** The original design retrieved relevant
    lesson content from Pinecone (a cloud vector DB) before answering; with
    no cloud vector store of any kind, the system prompt instead just names
    the linked lesson by title and otherwise relies on the model's own
    general subject knowledge. Less grounded in this platform's specific
    course material, by design, not by oversight.
  - **No sandbox-verification on escalate.** The original design ran any
    suggested fix through the Coding Sandbox before confirming it actually
    works, returning `sandbox_verified: true/false`. With no sandbox at
    all, escalation just returns the direct answer/fix with no verification
    claim attached.

Verified end to end against a local mock of Anthropic's `/v1/messages`
endpoint (so the request-building, response-parsing, quota, and
duplicate-detection logic could be tested for real without needing a real
API key) — real cost-per-message math (verified the `deep` tier correctly
costs exactly 3x the `fast` tier, matching the configured per-model rates),
quota exhaustion returning a real `429`, an upstream provider error
returning a graceful `503` instead of crashing, and cross-student
conversation access correctly 404ing rather than leaking another student's
chat. **Setting a real `AI_API_KEY` in `.env` is what makes this actually
answer real questions** — everything above is real, tested logic around
the one call that needs your own Anthropic credentials to go live.

One real bug found only by running this: `curl_init()` was undefined — the
`curl` PHP extension wasn't enabled in this machine's fresh `php.ini`
(same pattern as `mbstring`/`fileinfo` before it). Fixed locally by
enabling it; **`curl` ships enabled on virtually every PHP install,
GoDaddy included**, since so much PHP software depends on it — not
expected to be a real deployment risk, noted here for the same reason as
the others. A second fix: `AssignmentController::submit()` originally
re-stamped `submitted_at`/`is_late` on every call, so a retried request
(the doc explicitly calls out a flaky GoDaddy connection retrying this
exact endpoint) could silently move the recorded SLA timestamp forward.
Fixed with a state-machine guard — calling submit on an already-submitted
assignment now just returns the existing state unchanged, verified by
calling it twice in a row and confirming `submitted_at` didn't move.

## Phase 4e — Assessments (Exams), Project Lifecycle + Publishing, Progress Analytics

`docs/student-module/04e-apis-assessments-projects.md` covers three things;
all three needed real adaptation for the no-cloud/no-execution decisions:

- **Assessments** (`ExamController`): list/detail, start-or-resume an
  attempt (resumability — a second call while `in_progress` returns the
  same attempt, never a new one), per-attempt deterministic question
  shuffling (`mt_srand($attemptId)`, reseeded from system entropy
  immediately after — the same security-conscious pattern
  `PlacementController` already used, so nothing else in the request
  inherits a predictable sequence), autosave-per-answer, a cheating-flag log
  (blur/visibility events), submit, and result. **Server-side timer
  enforcement, not client-trusted**: the *next* request of any kind against
  an expired attempt auto-finalizes it first using whatever was already
  autosaved, exactly as the doc specifies — never a background job racing
  the client.
  Two real adaptations:
  - **No `POST /attempts/{id}/proctor-token` at all.** The original design
    reused Agora's token shape for proctored exams; with no Agora (or any
    video service) anywhere in this build, video proctoring doesn't exist.
    The cheating-flag log is the only proctoring signal. A live-monitored
    option could later reuse `ClassroomController`'s P2P mesh the same way
    a tiny classroom would — a real feature to design on purpose later, not
    something to fake here.
  - **`type='coding'` exam questions are never auto-graded by execution** —
    there is no sandbox in this build (confirmed: "no coding live should be
    there"). They route into the same manual-grading queue as
    `short_answer`/`long_answer`, leaving `marks_awarded`/`is_correct`
    explicitly `NULL` (not the column's `DEFAULT 0`) until a teacher grades
    them — `result()` reports `fully_graded: false` while any response is
    still pending.
- **Project Lifecycle + Publishing**: reuses the Assignments endpoints from
  04d (`type='project'`, combining `github_repo_url`/`demo_video_url`/
  `screenshots` at once, per the doc's explicit requirement), plus three new
  ones (`AssignmentController`): `GET /submissions/{id}`,
  `POST /submissions/{id}/publish-request` (creates `published_projects`
  with `is_public=0` — nothing reaches the public Achievement Showcase
  without an explicit mentor/admin approval, an Admin/Teacher-portal action;
  idempotent — a second call returns the same row rather than erroring),
  `GET /submissions/mine/publish-requests`.
  **The real adaptation**: submitting a project queues an "asynchronous"
  originality check, per the doc — the original design needed Pinecone
  (embedding similarity) plus a third-party plagiarism service, neither of
  which exist here. Reusing the one accepted external dependency (the AI
  Gateway), `cron/process-originality-checks.php` is a **GoDaddy cPanel Cron
  Job** (a real, ordinary shared-hosting feature — not a cloud service) that
  drains an `originality_check_queue` table every ~15 minutes: it asks the
  AI to rate how similar a submission looks to a handful of other students'
  submissions for the same assignment, and stores the result as
  `originality_score`. Cruder than real embedding-similarity, but real and
  running — advisory only, never an auto-reject gate, exactly as the doc
  requires.
- **Progress Analytics** (`ProgressController`, read-only): `snapshot`/
  `history`/`insights`. As the doc specifies, this only ever reads
  precomputed `student_progress_snapshots` rows — there's no live-
  aggregation path, by design (multi-table joins on every dashboard load is
  a real performance risk on shared hosting). `cron/compute-progress-
  snapshots.php` is the **other** GoDaddy cPanel Cron Job this phase adds
  (run nightly), computing attendance/completion/score percentages from the
  existing tables for every active enrollment.
  **The real adaptation**: `coding_success_rate` originally meant "share of
  sandbox executions with `exit_code=0`" — with no executions happening
  ever, there's nothing to measure that way. The cron instead computes it as
  the share of *graded* coding-type work (`type='code'` assignments +
  `type='coding'` exam questions) that scored at or above a passing bar — a
  grading-based "did it work" signal, the closest honest equivalent left.
  `coding_executions_count` stays `0` always, truthfully, since nothing in
  this build ever executes code.

Both cron jobs need to be registered in cPanel's **Cron Jobs** tool once
deployed — see `DEPLOYMENT.md`.

Two real bugs found only by running this:
1. `SELECT total_marks, passing_marks FROM exams e JOIN exam_attempts ea ...`
   crashed with "Column 'total_marks' in field list is ambiguous" — both
   tables have a column by that name. Fixed by qualifying `e.total_marks`.
2. The same NULL-vs-zero issue caught earlier in this project (`WalletController`,
   exam response grading) showed up again in the nightly snapshot cron:
   `attendance_percent` was computed as `0` whenever a student had *no*
   attendance rows at all (no sessions recorded yet), indistinguishable from
   a real, measured 0% attendance. Fixed by keeping the "no data" case
   explicitly `NULL` — verified by seeding one student with a real
   attendance row (correctly shows the real percentage) and leaving another
   with none (correctly shows `NULL`, not `0`).

Verified end to end: full exam attempt lifecycle including a question left
unanswered (correctly excluded from the marks denominator) and a coding
question correctly left pending for manual grading; `max_attempts_reached`
enforcement; project submission → grading → publish-request (including the
idempotent-retry and cross-student-404 cases) → the originality cron
actually calling the (mocked) AI and writing back two *different* scores for
two different submissions; and the nightly snapshot cron producing correct
numbers cross-checked by hand against the seeded data.

## Phase 4f — AI Risk Detection (parent-facing slice), Parent Visibility, Monthly Parent Reports, Payments/Refunds/Freeze

`docs/student-module/04f-apis-parent-billing.md` is wired up (`ParentController`,
`PaymentController`):

- **Risk Visibility**: `GET /parent/students/{id}/risk-summary` — a softened
  advisory view derived from `risk_scores.intervention_status`, never the
  raw `score_value`/`contributing_factors` (the doc's explicit reasoning: a
  numeric "dropout risk: 78" shown raw to a parent would do more harm than
  good without a human framing it). No real adaptation needed — this was
  always a pure-SQL read, no cloud dependency in the original design either.
- **Parent Visibility**: `children`, `dashboard` (reads the same snapshot as
  04e), `attendance`/`recordings` (scoped proxies of 04c's student-facing
  data), and `wallet` (the one parent view *never* served from a
  precomputed snapshot — read live every time, since stale financial data
  shown to a parent is a trust problem the doc explicitly calls out).
  Every endpoint re-checks `consent_status='granted'` **and** the relevant
  `can_view_*` boolean on every single request — verified by revoking
  `can_view_billing` mid-session and confirming the very next wallet call
  403s while `risk-summary` (deliberately ungated by any `can_view_*`

  calls out: once a student's computed age (from `users.date_of_birth`)

  that student 403s with `student_now_adult_consent_required` instead of
  silently continuing on stale minor-era consent — verified by bumping a
  seeded student's age past 18 and confirming the cutoff fires, then
  restoring it and confirming access returns.
- **Monthly Parent Reports**: list/detail/mark-viewed (`POST .../viewed` is
  a one-way flag — an already-viewed report ignores a repeat call, verified
  by calling it twice and confirming the timestamp doesn't move on the
  second call). **The real adaptation**: the doc's `pdf_url` example points
  at an S3 object; this build has no S3 (or any cloud storage) at all. Two
  things had to exist that the API doc doesn't itself specify, since
  otherwise `parent_reports` would simply never have any rows to read:
  - **`App\Core\SimplePdf`** — a minimal, hand-rolled PDF writer (single
    font, word-wrap, multi-page pagination, no external library, no
    Composer dependency — matching this app's existing no-Composer-at-
    runtime convention). Verified for real: generated PDFs were parsed back
    with an independent Python PDF library (`pypdf`), confirming both a
    single-page and a forced multi-page report extract back byte-for-byte
    correct text in the right order.
  - **`cron/generate-parent-reports.php`** — a third GoDaddy cPanel Cron Job
    (monthly), pulling the same numbers as the 04e progress snapshot, asking
    the AI Gateway for a short parent-friendly summary, and writing the PDF
    to local disk. `pdf_url` now stores a relative path (the same convention
    as Materials/Recordings' `file_path`), signed fresh on every
    `GET /reports/{id}` call via `App\Core\FileStorage` — never a stored,
    eventually-expiring URL.
- **Payments/Refunds/Freeze**: payment history/detail/retry (a retry is a
  fresh `payments` row re-entering the same checkout flow, never mutating
  the original failed row — verified: retrying succeeds once, then 422s
  with `not_failed` on the original since it's no longer in `failed`
  status), `refund-request` (opens a `support_tickets` row, never an
  immediate refund — an admin reviews against policy first), and
  `freeze-history` (the full audit trail from `wallet_freeze_log`).

Two real bugs found only by running this:
1. **Non-ASCII text silently corrupted in generated PDFs.** PDF text
   strings for a standard, non-embedded font like Helvetica are single-byte
   WinAnsi, not UTF-8 — writing raw UTF-8 bytes (an em-dash in this app's
   own report heading, and routinely in AI-generated summary text: smart
   quotes, em-dashes) produced mojibake with no error or warning, the kind
   of bug that's easy to ship without ever seeing an exception. Fixed in
   `SimplePdf::escape()` by transliterating the common offenders to ASCII
   and dropping anything else outside WinAnsi's range — verified by
   regenerating a report and confirming the heading's em-dash now renders
   as a plain hyphen instead of garbage bytes.
2. A zombie test MySQL process from an earlier verification round was still
   running and not actually bound to any port — caught when a later test
   round's connection attempt failed. Resolved by identifying it
   conclusively (zero listening sockets, vs. the real `MySQL80` service's
   confirmed listening socket on its default port 3306) before terminating
   only that orphaned process; `MySQL80` was independently reconfirmed
   `Running` immediately before and after.

## Phase 4g — Course Completion, Certificates, Renewal/Upsell, Referrals, Support System

`docs/student-module/04g-apis-completion-growth.md` is wired up
(`EnrollmentController`, `CertificateController`, `RecommendationController`,
`ReferralController`, `SupportController`):

- **Course Completion**: read-only — `App\Core\CourseCompletion::evaluate()`
  is the single shared source of truth for the five criteria, used by both
  the `completion-requirements` endpoint and the completion cron, so the two
  can never silently disagree about what "done" means. The doc describes
  completion as an "event-driven check," but on shared hosting with no
  queue/worker process and no way for this app to be notified the instant
  an Admin-portal teacher grades something, `cron/check-course-completion.php`
  (hourly) is the honest equivalent. Two criteria have no precise schema
  signal to check (no `is_capstone` flag, no dispute-tracking table) —
  resolved with the simplest defensible interpretation, documented in
  `CourseCompletion`'s own docblock rather than guessed silently.
  `grace_period_ends_at` is computed at read time (no stored column) and is
  correctly `null` once every requirement is actually met, even in the
  window before the hourly cron has processed it — verified by hitting this
  exact race directly: a student whose criteria were all already
  satisfied showed a grace-period date until this was fixed.
- **Certificates**: `certificates.pdf_path` already existed in the real
  schema but **no PDF-generation code exists anywhere in the Admin panel**
  (confirmed by reading `SuperAdmin\CertificateController::issue()` —
  it creates the DB row and stops). `App\Core\CertificateGenerator` (built
  on `SimplePdf`, now with basic centered/variable-size text support for a
  real certificate layout) is the first real producer of that file.
  Deliberately does **not** render `certificate_templates.html_template`/
  `css_styles` — an HTML+CSS-to-PDF engine needs a headless browser or a
  heavy library, neither available without Composer or a cloud service;
  that template system stays unused, same as the dropped Coding Sandbox
  tables. `confirm-certificate-name` writes to a new
  `enrollments.confirmed_certificate_name` column (another gap: no column
  existed anywhere to record this). Auto-issuance matches the Admin panel's
  own manual `issue()` certificate-number/verification-code format exactly,
  so a cron-issued and a teacher-issued certificate are indistinguishable.
  Verified end to end: a fully-qualifying seeded student was auto-completed
  and auto-issued a real certificate by the cron, the generated PDF was
  independently parsed back with `pypdf` confirming correct text, downloaded
  through the same signed-URL mechanism as Materials/Recordings, and
  `reissue-request` regenerated the same `certificate_number` rather than
  creating a second one.
- **Renewal/Upsell**: `GET /recommendations` — the ranking mechanics are an
  explicit "Phase 5" AI concern per the doc; this endpoint only ever serves
  whatever's already in `course_recommendations`, including the graceful
  `explore_other_tracks` no-fit state, and sets `shown_at` as a side effect
  of being read (same "the request is the access event" pattern as
  material downloads).
- **Referral System**: lazy code generation on first request (most users
  never visit this screen) — verified idempotent (a second call returns
  the identical code, not a new one). No "create a referral" endpoint by
  design; a `?ref=` lead is ordinary lead-attribution metadata, not a
  parallel funnel.
- **Support System**: categories/tickets/replies/satisfaction against the
  existing Admin-panel tables. `ticket_replies` rows with
  `is_internal_note=1` are unconditionally filtered out of the student-
  facing response regardless of who's asking — verified by seeding one and
  confirming it never appears in `GET /support/tickets/{id}`.
  `POST .../satisfaction` correctly 422s on a still-open ticket and
  succeeds once `status` is `resolved`.

Two real bugs found only by running this:
1. **`grace_period_ends_at` shown even when every requirement was already
   met** — the original check only looked at the enrollment's `status`
   (still `active`, since the hourly cron hadn't run yet) and the batch's
   end date, never the requirements themselves. Fixed by threading
   `CourseCompletion::evaluate()`'s `all_met` result into the grace-period
   decision — a student who's already finished never sees a grace-period
   date, regardless of whether the cron has caught up yet.
2. A leaked test certificate PDF (generated during this round's
   verification) was found sitting in `storage/app/certificates/` before
   packaging — removed before zipping, same class of mistake as the leaked
   parent-report PDF in the previous phase, now something to specifically
   check for before every deploy-zip regeneration.

## Phase 4i — Live Quizzes, Code Replay, Offline Access, Calendar Sync

`docs/student-module/04i-apis-quizzes-replay-offline.md` covers four
things; one is skipped entirely, the other three needed real adaptation
(`LiveQuizController`, `OfflineDownloadController`, `CalendarController`,
plus a `syncOffline()` addition to `LessonController`):

- **Code Replay is not implemented at all.** This isn't a "swap S3 for
  local storage" question like everywhere else in this build —
  `code_replay_sessions.workspace_id` is a **required** foreign key to
  `code_workspaces`, a table nothing in this codebase ever populates,
  since live code execution was dropped entirely (confirmed: "no coding
  live should be there"). There is no workspace, no live coding session,
  nothing to replay. Building this API would mean a hollow shell
  referencing a table that can never have rows — skipped, with the reason
  stated plainly rather than building dead code around a missing feature.
- **Live Quizzes**: the doc frames launch/leaderboard/close as "pushed
  over the live class's existing Pusher/Ably channel, never polled" — no
  Pusher/Ably exists anywhere in this build. `GET /classes/{id}/live-quiz/current`
  is the real adaptation: clients poll it the same ~2s cadence already used
  for chat/signal polling, discovering a newly-launched quiz the same way
  they discover a new chat message. `live_quizzes` rows themselves are
  created by teacher-side Admin-portal code entirely outside this
  codebase's reach — this controller only reads/responds to quizzes that
  already exist. `response_time_ms` is computed server-side from
  `launched_at`, never client-reported, and a `rapid_quiz` correctly scores
  10–20 points based on how fast the (server-measured) response was. Two
  schema gaps filled: `ad_hoc_correct_answer` (a teacher-typed-on-the-fly
  question had nowhere to record which option is right) and
  `live_quiz_responses.explain_mode_conversation_id` (the doc says this
  "populates once the client does so" — but never had a column to populate).
  `explain-mode` reuses the existing AI Doubt Solver rather than inventing
  a parallel explanation feature, exactly as the doc requires.
- **Offline Access / Download Mode** — **the most significant adaptation
  in this catalog.** The original design is AES-128 HLS-encrypted video
  with a server-issued `decryption_key` per validation call: a real
  DRM-shaped pipeline needing infrastructure well beyond hand-rolling on
  shared hosting, and this build never even has an HLS pipeline to begin
  with (recordings are already a single progressively-downloaded file, an
  earlier, separate scope cut). What survives intact: the *access-control*
  shape — `download_token`, bounded `expires_at`, periodic re-validation,
  student-initiated revoke, account-sharing flagging (verified by
  requesting downloads from 4+ distinct device fingerprints in one day and
  confirming each still succeeds while a review-flag gets logged — never
  auto-blocked, exactly as the doc requires). Only the cryptographic
  payload is gone: `validate()` returns a freshly-signed local-file URL
  (the same `FileStorage` mechanism as Materials/Recordings) instead of a
  `decryption_key` for a file that was never actually encrypted.
  `content_type='note_export'` generates a real PDF on demand via the same
  `SimplePdf` writer used for parent reports and certificates — a note has
  no file at all until offline access asks for one.
  `POST /lesson-progress/sync-offline` (on `LessonController`) reuses the
  exact same completion-threshold logic as live progress updates, but
  using the client-recorded `recorded_at` — and is monotonic: an offline
  event can never regress progress that's already more advanced, verified
  by sending a smaller, older progress value after a larger one and
  confirming it's correctly ignored (`events_applied: 0`).
- **Calendar Integration**: `GET /calendar/providers` (real — reads actual
  `calendar_connections` rows) and `DELETE /calendar/connections/{id}`
  (real — a plain row deletion) are built. **`connect`/`reconnect` (the
  actual OAuth2 dance) are not implemented** — both need real developer
  credentials from Google/Microsoft (plus a CalDAV client for Apple), which
  is a separate undertaking requiring the user to first register apps with
  three different providers, unlike the AI Gateway's single API key. Half-
  building this with placeholder client IDs would produce something that
  looks complete but can't actually authenticate anyone — deferred and
  documented instead.

One real bug found only by running this: `LiveQuizController` had **no
batch-membership check at all** — any authenticated student could respond
to, view, or see results for *any* live quiz, not just ones for classes
they're actually in. Fixed by folding a `batch_students` join into
`requireQuiz()` (the same 404-not-403 "don't confirm it exists" pattern
used throughout this catalog) — verified with a student who was never in
the relevant batch, confirming a clean `404` instead of unrestricted access.

## Phase 5a — AI Gateway Architecture, RAG Retrieval, Cost Control

`docs/student-module/05a-ai-gateway-rag-cost-control.md` is the cross-
cutting AI infrastructure every AI-calling feature relies on. Most of it
was already settled by earlier no-cloud decisions and needed no new work
(AWS Bedrock hosting-duality doesn't apply — this build only ever has the
one GoDaddy-only hosting target; Pinecone RAG was already dropped entirely
in 04d). What's real and new in this pass (`App\Core\AiGateway`/`AiQuota`/
`AiUsageLog`/`Moderation`, `config/ai_tiers.php`):

- **Quota enforcement, now actually universal.** 05a is explicit that
  *every* feature calling the Gateway shares the same quota gate — but the
  atomic check-and-reserve logic only ever lived inside `AiController`.
  Notebook AI (summarize/flashcards/quiz-generation, recording→notes) was
  **unmetered** before this pass — a real gap, not a hypothetical one.
  Extracted into `App\Core\AiQuota`, now used by every AI-calling
  controller identically.
- **Moderation, built for the first time.** Nothing moderated AI input or
  output anywhere in this build before now, despite minors being users.
  05a's design is two LLM-backed passes including per-chunk streaming
  moderation — adapted down to what's actually consistent with this
  build's own earlier decision to drop streaming entirely: a free,
  rule-based keyword precheck before any provider call (catches the
  obvious cases without doubling the cost of every single AI interaction
  with a second paid call), plus a safety instruction folded into every
  system prompt for nuance the keyword list can't catch, plus the same
  check applied to the complete response before it's returned. Verified by
  sending a blocked phrase and confirming it never reaches the provider or
  spends quota.
- **Platform-wide daily spend circuit breaker, built for the first time.**
  Protects against a runaway bug (an infinite retry loop, a repeatedly
  oversized response) — distinct from any individual student's quota,
  which doesn't catch that. 05a says this "pages ops if approached"; with
  no alerting channel in this build, refusing further calls plus a
  `CRITICAL` log line is the honest equivalent — a log nobody's paged for
  isn't real protection. Verified by temporarily lowering the cap and
  confirming calls are actually refused, with the exact spend numbers in
  the log.
- **One retry on transient failure.** 05a's "Failover" describes retrying
  against a fallback provider/route — this build only ever has the one
  accepted API, so there's no second route to fail over to. What's real
  and buildable instead: one retry against the *same* route on a transport
  failure or 5xx, never on a 4xx (retrying a deterministic error wastes a
  call for nothing). Verified two ways: a transient failure that succeeds
  on retry, and a permanent failure that still fails cleanly afterward (not
  looping).
- **Model tiering as config**, matching 05a's own table exactly
  (`config/ai_tiers.php`, `{feature}.{mode} → fast|deep`) — a tier is now a
  config change, not a per-call decision scattered across controllers. This
  also **corrected** a real mismatch: Notebook summarize/flashcards were
  hardcoded to the fast tier; 05a's table says both should be deep
  ("quality of the extracted content matters more than speed; not on a
  synchronous user-waiting path"), now fixed.
- **Prompt versioning, lightly.** `ai_messages.model_used` (and the new
  `ai_usage_log.model_used`) now records a composite like
  `claude-sonnet-4-6@doubt_solver_hint_v1`, matching 05a's stated format,
  rather than just the bare model name.
- **A schema gap, found while wiring quota universally**:
  `ai_messages.conversation_id` is `NOT NULL`, but Notebook AI and the
  originality-check/parent-report crons are one-shot calls with no
  conversation to attach to — without somewhere to log them, their real
  cost would be invisible to both per-student reporting and the new
  circuit breaker. `ai_usage_log` is the gap-fill; the circuit breaker sums
  both tables.

RAG retrieval (§2) and the AWS Bedrock transport path (§1's "hosting
duality") are not implemented, not as an oversight but because they're
exactly the cloud infrastructure this entire build has avoided from the
start — already covered by the "No-cloud architecture decisions" table.

## Phase 5b — Doubt Solver & Coding Assistant Prompt Design

`docs/student-module/05b-ai-doubt-solver-coding-assistant-prompts.md` is
specifically the prompt templates and call orchestration layered on top of
05a's shared Gateway/quota/moderation plumbing — most of it was already
satisfied by `AiController`'s existing implementation once 05a's pass was
done. What changed in this pass (`AiController::buildSystemPrompt()` and
its new `baseLayer()`/`modeLayer()`/`escalationLayer()` helpers):

- **The prompt is now explicitly layered**, matching 05b §1's structure
  (base persona/safety layer, then a mode-specific or escalation layer) —
  previously a single concatenated string built ad hoc. The RAG context
  layer is the one documented layer that's still skipped entirely, for the
  same reason as everywhere else in this build: no vector DB exists.
- **The exact core directive is now in the base layer**: *"You teach; you
  do not hand over direct answers unless explicitly told you may."* —
  05b's stated anchor sentence every mode-specific instruction builds on
  top of, not just implied by the individual mode instructions.
- **A coding-assistant-specific length-guard message.** The length/line-
  count threshold itself already existed (checked before any model call,
  zero tokens spent on the canned response) — what was missing was 05b
  §3's specific framing for a code dump ("let's focus on one function")
  versus the Doubt Solver's framing for a rambling question ("let's focus
  on one question at a time"). Verified both: a long code paste in a
  `coding_assistant` conversation gets the function-focused message, the
  identical-length text in a `doubt_solver` conversation gets the
  question-focused one.
- **The sandbox-verification loop (05b §3) is not implemented** — 05b
  calls this "the one place that needs a concrete mechanism, not just
  careful prompting": propose a fix, run it against a throwaway workspace
  copy, confirm or hedge based on the real outcome. There is no sandbox in
  this build at all, so there is nothing to run a proposed fix against.
  Every `debug`-mode response is now framed with appropriate uncertainty
  baked directly into the mode layer instead ("you cannot run code to
  verify a fix in this conversation, so phrase any suggested fix as
  something to try, not a confirmed solution") — an honest, permanent
  hedge rather than a confident "verified" claim this build can never
  actually back up.
- **Coding Assistant context is the student's pasted code/error, not live
  `workspace_files`** — 05b §3 describes bundling every file in the
  workspace as context; with no workspace/sandbox at all, there's nothing
  to bundle. Already true before this pass, now stated explicitly in
  `AiController`'s class docblock rather than left implicit.

Everything else in 05b — hint mode's "don't capitulate on a second direct
ask," practice mode's "analogous, not a reskin" instruction, near-duplicate
detection never being mentioned to the model, escalation being a genuinely
distinct prompt rather than a flag — was already correctly implemented
during 04d/05a and needed no change; this pass's job was auditing that
against 05b's specifics and fixing the few places that were genuinely off,
not rebuilding what already matched.

## Phase 5c — Notebook AI Prompt Design, Risk/Report Narrative Generation

`docs/student-module/05c-ai-notebook-risk-report-prompts.md` surfaced real,
previously-missing functionality, not just prompt wording:

- **Class-note generation is now a real map-reduce**, not a single
  truncated pass. The previous implementation cut `transcript_text` off at
  12,000 characters (`mb_substr(..., 0, 12000)`) — a real 60-90 minute
  class transcript silently lost everything past that point. Now: the
  transcript is split into up to 9 chunks, each summarized independently on
  the fast tier with an approximate `[MM:SS]` timestamp (the "map"), then
  one deep-tier call reorganizes every chunk's bullets **by topic, not by
  chunk boundary** while preserving the timestamp tags (the "reduce") —
  verified end to end with a real ~18,000-character transcript, confirming
  exactly 6 map calls (matching the chunking math) followed by 1 reduce
  call, with the final note correctly organized by topic rather than dumped
  as disconnected per-chunk lists. The timestamps are a stated
  approximation (proportional to chunk position × `duration_seconds`), not
  real per-word timing — there's no actual speech-to-text pipeline in this
  build to provide that.
- **Note summarize/flashcards prompts now match the doc's exact wording**
  (conceptual framing instead of a generic restatement; "test understanding
  of a concept, not recall of an exact sentence"; the trivial-card
  exclusion; the cap corrected from 10 to 8 per the doc's explicit number).
- **Risk scoring, built for the first time.** `risk_scores` existed in the
  schema and `ParentController::riskSummary()` already read from it, but
  nothing ever populated it before this pass — `cron/compute-risk-scores.php`
  is the first real producer. 05c §4 states a deliberate, restated-in-code
  boundary: **the score itself is always deterministic arithmetic
  (attendance-trend decline, assignment-completion rate, login recency) —
  never an LLM** — a score that triggers real interventions has to be
  exactly reproducible, which free-text model judgment can't guarantee.
  `intervention_status` is a computed escalation *tier* matching exactly
  what `ParentController::riskSummary()` already checks for, not a literal
  claim that a WhatsApp/email/call went out (the Communication Engine that
  would actually send one isn't built in this pass).
- **Monthly Parent Report narrative, substantially restructured.** The
  previous version was one free-flowing 2-4 sentence paragraph with no
  grounding instruction and no risk-score awareness at all. Now: an
  explicit anti-hallucination grounding instruction ("use only the data
  provided... do not invent specific events, dates, or numbers"), three
  labeled sections (Strengths / Areas for growth / Recommended next steps)
  parsed out of the response and rendered as separate headings in the PDF
  — verified by inspecting the actual generated PDF and confirming all
  three sections render distinctly, not as one undifferentiated block —
  and an elevated risk signal (read from the new `risk_scores` table) is
  now narrated softly and tied to a concrete next step, *never* as a raw
  number, the same advisory-only principle `ParentController::riskSummary()`
  already applies on the parent-dashboard side. The partial-period framing
  instruction was also tightened to match the doc's exact "frame as their
  first X days" wording.

One real bug found only by running this: the new risk-scoring cron's
attendance-decline calculation queried by `batch_id` alone — averaging
attendance across **every student in the batch**, not the individual
student being scored. Caught immediately by seeding two students with
deliberately different attendance patterns in the same batch and noticing
both got an identical (wrong) `recent_attendance_pct`/`lifetime_attendance_pct`.
Fixed by also filtering on `student_id`, verified by re-running and
confirming the two students now get correctly different scores (a
declining-attendance, stale-login student scored 97/`parent_escalated`; a
consistently-attending, just-logged-in student scored 0/`none`).

`05c §1`'s voice-to-text cleanup pass is not implemented — it runs "after
transcription," and there is no transcription step in this build at all
(`NoteController`'s docblock already covers why `voice-transcribe` was
dropped). `05c §3`'s quiz-from-notes design (private, `source_note_id`-
linked, skipping the shared question-bank review gate) was already exactly
how `NoteController::generateQuiz()` worked from 04h — confirmed correct
against the doc, no change needed.

## Phase 5d — Placement Scoring, Originality Self/Other-Similarity, Course Recommendation Engine

`docs/student-module/05d-ai-placement-originality-recommendations.md` —
the document that explicitly **closes out Delivery Phase 5 — AI Workflows
in full**. All three sub-parts implement the same recurring theme this
document names directly: every consequential decision (a placement level,
a plagiarism flag, a recommended course) is made by deterministic logic
over model-produced inputs — the model scores or narrates, a rule decides.

- **Placement communication-section scoring, wired up for real.**
  `PlacementController` had deferred this since before the AI Gateway
  existed (`communication_score` was hardcoded `null`, "pending a real AI
  Gateway or manual mentor scoring"). Now that one exists (04d/05a),
  `scoreCommunication()` runs a rubric-based fast-tier call returning a
  0-100 score *plus a short rationale* (stored in a new
  `communication_score_rationale` column — no column existed to hold it
  before this pass, the same transparency principle `risk_scores.contributing_factors`
  already established). Deliberately **not gated by the per-student
  `AiQuota`** — placement is a mandatory flow a student can't choose to
  skip, so blocking it behind an unrelated daily chat-help cap would be
  wrong; cost is still logged via `AiUsageLog` for the platform-wide
  circuit breaker. `recommended_level` now averages in the real
  communication score instead of just coding+logical. Borderline routing
  (05d's explicit requirement: a deterministic threshold check, never the
  model self-reporting uncertainty) is computed in `isBorderline()` — a
  configured ±5 band around either level cutoff — and surfaced as
  `is_borderline` on `result()`'s response. There's no mentor "mini-check"
  queue UI to route it to (that's the separate Admin panel, out of this
  codebase's scope); the signal is real and exposed at the API level, the
  same honest boundary used everywhere else a mentor-side action is
  referenced but not built here. Verified end to end: a real placement
  attempt (MCQ + true/false + a short-answer communication question) was
  submitted, scored by the mock LLM, and correctly averaged into
  `recommended_level`; `isBorderline()` was also verified directly against
  threshold/non-threshold score combinations.
- **Originality check now runs a genuine self-similarity pass**, separate
  from the existing other-similarity (plagiarism) pass — 05d's explicit
  false-positive case is "a student legitimately building on their own
  earlier work," which a single undifferentiated similarity number would
  punish. Self-similarity (against the *same* student's own prior
  submissions) is logged informationally and never touches
  `originality_score`; only the comparison against *other* students'
  submissions to the same assignment does. The self-similarity call is
  wrapped in its own try/catch so a failure there can never fail the queue
  item or block the actual plagiarism-relevant check. The doc's second
  originally-designed signal — a MOSS-style public-code plagiarism
  service — is **not implemented**: that's a distinct third-party SaaS
  requiring its own account, not the "an API key, no infrastructure"
  shape of this build's one accepted cloud exception. Verified by seeding
  a student with two submissions to *different* assignments (since
  `assignment_submissions` has a `UNIQUE(assignment_id, student_id)`
  constraint — a student can't submit the same assignment twice, so the
  self-similarity test data had to use a separate assignment) and
  confirming both passes ran, logged separately and distinguishably via
  each call's `model_used` prompt-key suffix, with only the other-similarity
  pass affecting `originality_score`.
- **Course recommendation engine, built for the first time.**
  `RecommendationController::index()` already existed and explicitly only
  ever served whatever the (nonexistent) engine had computed — nothing
  populated `course_recommendations` before this pass, the same shape of
  gap as risk scoring in `05c`. `cron/generate-course-recommendations.php`
  implements 05d's exact two-stage design: Stage 1 (deterministic, no
  model call) reads a new, admin-curated `course_next_steps` table
  (curriculum mapping — empty by default; no Admin-panel UI exists yet to
  manage it, documented in `DEPLOYMENT.md` as a direct-SQL step until one
  does) filtered by the student's performance bucket
  (`student_progress_snapshots`) and `student_profiles.interests`,
  producing `confidence_score` entirely in application code — never asked
  of the model, keeping the one number this feature's real-world accuracy
  gets measured by (via `shown_at`/`converted_at`) out of the LLM's hands.
  Stage 2 (fast-tier) only narrates an already-decided pick. The cron also
  closes the loop `RecommendationController`'s own comment left open
  ("`converted_at`... set by a backend job, never a client call") by
  detecting a student enrolling in a previously-recommended course.
  Verified end to end: a completed enrollment with a real
  `course_next_steps` mapping correctly produced ranked, confidence-scored
  recommendations (interest-tag matches correctly boosted confidence,
  verified against the exact expected arithmetic); a completed enrollment
  with *no* mapping correctly produced zero recommendations rather than an
  error (the `explore_other_tracks` graceful state, confirmed via the read
  endpoint); reading the recommendations correctly set `shown_at`; and
  simulating an enrollment into a shown recommendation's course correctly
  set `converted_at` on the next cron run while leaving an un-acted-on
  sibling recommendation untouched.

No bugs were found in the new production code itself this round — the one
issue hit during testing was a test-seed gap (a `student_profiles` row
missing for a seeded student, so an interest-match confidence boost
appeared not to apply on the first run), correctly diagnosed by checking
the table directly rather than mistakenly "fixing" the cron's matching
logic, then re-verified correct once the seed was fixed.

**Delivery Phase 5 — AI Workflows is now complete in full** (`05a`
through `05d`): the shared Gateway/quota/moderation/cost infrastructure,
the Doubt Solver/Coding Assistant prompts, Notebook AI plus risk/report
narrative generation, and this phase's placement/originality/recommendation
features. Every AI-touching workflow named across the original 35-document
series now has a real, tested implementation or an explicitly documented,
reasoned reason it doesn't (the sandbox-verification loop, voice
transcription, and the MOSS-style plagiarism service — each requiring
infrastructure or a second external dependency this no-cloud,
single-AI-exception build deliberately doesn't have).

## Phase 6 — Communication Orchestration Engine

`docs/student-module/06-communication-engine.md` — the multi-channel
dispatch layer every workflow phase since `03a` referenced in passing as
"Phase 6" without it existing yet. `communication_logs` and `domain_events`
already existed in the schema (an earlier phase anticipated them) but
nothing wrote to either — the same "table exists, nothing produces it"
shape as several earlier gaps in this build.

**`App\Core\Notifier::send(userId, triggerEvent, context)`** is the one
dispatch entry point the doc calls for — every wired trigger in this phase
calls this rather than hand-rolling its own send logic. What it actually does:

- **Real for `in_app` and `email`; explicitly not implemented for
  `whatsapp`/`sms`.** In-app writes directly to the Admin panel's own
  existing `notifications` table (shared, not duplicated — this app just
  had no read endpoint for it before this phase, see below). Email sends
  for real over raw SMTP via a new hand-rolled client, **`App\Core\SimpleMailer`**
  (same "no external library" instinct as `App\Core\SimplePdf`, applied to
  mail) — talking to whatever's configured in `config/mail.php`, which on
  GoDaddy is the cPanel mailbox already included in the hosting plan, not a
  third-party cloud add-on. WhatsApp and SMS are real third-party cloud
  accounts (Meta's WhatsApp Cloud API, a Twilio/MSG91-style SMS gateway) —
  outside this build's one accepted cloud exception (a pay-per-call AI
  API) — so `dispatchChannel()` always returns a real, honestly-logged
  failure for those two rather than faking a delivery that never happened.
  This isn't a dead end: it's what correctly exercises the **fallback
  mechanic** for real, verified end to end — a WhatsApp attempt genuinely
  fails with a clear `failed_reason`, and the dispatcher genuinely moves to
  the next configured channel (email), which genuinely sends.
- **Idempotency** (06 §2): before any send, checks for an existing
  `communication_logs` row (or a still-`pending` `notification_queue` row)
  matching the exact `trigger_event` string. Per-occurrence triggers
  disambiguate by appending `:{entityId}` to the trigger_event itself (e.g.
  `badge_earned:3`, `assignment_reminder_24h:482`) — the schema has no
  separate entity-id column, so this is the literal mechanism 06 §2's own
  naming convention describes ("distinguishable instances get their own
  trigger_event string"). Verified by awarding the same badge twice in a
  row and confirming the second award produced zero new notification rows.
- **Channel selection & fallback** (06 §3): `config/notification_channels.php`
  configures, per trigger, an ordered external-channel sequence and whether
  it's `'fallback'` (try in order, stop at first success) or `'all'` (send
  every listed channel regardless of the others — the doc's "+" notation,
  e.g. the monthly parent report going out by both email and WhatsApp).
  `in_app` is never part of that sequence — it fires unconditionally on
  every send, the floor 06 §3 guarantees regardless of any external-channel
  opt-out in `users.notification_preferences` (checked before the
  trigger's default sequence is applied, per-channel, defaulting to
  allowed when a channel key is simply absent — no shape for this JSON
  column existed anywhere in the codebase before this phase, since nothing
  had ever read or written it).
- **Escalation + family batching** (06 §4): `notification_queue` is a new
  gap-fill table — any trigger configured `batchable => true` writes here
  instead of sending immediately. `cron/process-notification-batches.php`
  drains it, grouping by a `batch_key` when one was supplied (the
  family-level case 06 §4 explicitly separates from per-student batching)
  or by `user_id` otherwise, and combines each group into one consolidated
  send via `Notifier::dispatchBatch()`. Verified two ways: (1) two
  different children's `monthly_parent_report_ready` rows, both written
  with the same parent's `batch_key`, correctly combined into one in-app
  notification and one real email listing both children by name — while
  the two underlying `parent_reports` documents stayed completely separate
  rows, never merged (06 §4's explicit point); (2) a risk-tier escalation
  correctly combined and fell back from WhatsApp (failed, documented
  reason) to email (sent) in one combined dispatch.
- **Cadence definitions** (06 §6): `config/cadences.php` is a declarative
  registry; **one** generalized cron, `cron/process-cadences.php`,
  evaluates every registered cadence each tick rather than one bespoke job
  per feature. The one cadence implemented this pass — assignment
  reminders (`_24h`/`_6h`/`_overdue`, anchored on `assignments.due_date`,
  or `assignment_submissions.extended_due_date` when set) — was verified
  against a real overdue assignment and a real due-soon one: exactly the
  offsets that had actually crossed fired (1 of 3 for the due-soon one, all
  3 for the overdue one), and a student who'd already submitted was
  correctly excluded by the anchor query itself, not just by idempotency.
  The evaluator deliberately checks `now() >= anchor + offset` with no
  tight "just crossed" window, relying entirely on `Notifier`'s own
  idempotency to make repeated, at-least-once cron execution safe (06 §2) —
  this also means a cron outage of any length is caught up on the next
  run, never silently missed.
- **Templates** (06 §5): `config/notification_templates.php`, versioned the
  same way AI prompts are — `communication_logs.template_used` records
  exactly which version rendered. Per-channel variants of the same
  trigger_event, localized by `users.language` with a fallback to `en`.
- **The `domain_events` outbox, Student-app side** (06 §7):
  `cron/process-domain-events.php` polls `WHERE processed_by_student_app = 0`,
  applies the effect, and flips only its own flag — never
  `processed_by_admin`, verified directly (it stayed `0` after the
  Student-app consumer ran). Only `teacher.reassigned` is handled — 06 §7's
  own worked example — calling `Notifier::send(..., 'batch_reassignment_notice', ...)`
  for every student in the reassigned batch. An unhandled event type
  **throws rather than just logging**, a real bug caught and fixed during
  this pass: the first version logged a warning in the `default` match arm
  but didn't throw, so the caller flipped `processed_by_student_app = 1`
  anyway — silently marking an event "handled" when nothing had actually
  applied it. Fixed by throwing instead, verified by inserting an
  unhandled-type event and confirming it stayed unprocessed. There is no
  Admin-side code yet that *writes* a `teacher.reassigned` row (a change to
  the separate, pre-existing Admin panel codebase, out of this pass's
  scope) — tested here by inserting a row directly to simulate an
  Admin-originated event, the correct way to test one half of an outbox
  integration point without needing the other side's producer to exist yet.
- **A new student/parent-facing read endpoint**, `NotificationController`
  (`GET /notifications`, mark-read, mark-all-read, delete) — the Admin
  panel already has its own equivalent reading the same shared
  `notifications` table, but it's a separate PHP application with its own
  auth; a student or parent authenticated against *this* API had no way to
  reach it. Verified end to end, including `unread_count` dropping after
  mark-read/mark-all-read.

**Five representative triggers were wired into already-built producers**
(not the full 06 §6 catalog — registering a new one is the same pattern,
just not done for every entry this pass): `badge_earned`
(`App\Core\Gamification`), `completion_notification`
(`cron/check-course-completion.php`), `monthly_parent_report_ready`
(`cron/generate-parent-reports.php`, recipient correctly resolved to the
*parent* via `parent_student_links`, not the student the report is about),
`risk_tier_nudge`/`risk_tier_mentor_call`/`risk_tier_parent_escalation`
(`cron/compute-risk-scores.php` — 06's `risk_tier1_nudge`...`risk_tier5_escalation`
collapse onto this build's 3 real `intervention_status` values, since
there's no 5-level granularity in the schema to map onto), plus
`assignment_reminder_*` via the cadence engine and `batch_reassignment_notice`
via the domain_events consumer.

One more real bug found while wiring `risk_tier_mentor_call`'s templates:
the in-app variant addressed the student directly ("a mentor will be
reaching out to **you**"), but the WhatsApp/email variants referenced
`{student_name}` in the third person, as if addressed to a parent —
inconsistent, since `Notifier::dispatch()` sends every channel of one
trigger to the *same* recipient. Fixed by making all three channel
variants consistently student-facing, since this tier's actual recipient
(by design, distinct from `risk_tier_parent_escalation`) is the student.

## Phase 7 — Scaling Strategy

`docs/student-module/07-scaling-strategy.md` is a **decision framework for
a future the standing 3-year project mandate explicitly rules out** —
"don't assume cloud deployment will come in the future" was the governing
instruction from day one of this whole build. Most of the doc's actual
content (GoDaddy → AWS migration, Redis, Pinecone → pgvector, Piston →
Firecracker, Agora → self-hosted SFU) is **deliberately not implemented
here** — building toward any of those would contradict the mandate this
entire codebase was built under, not extend it. That content is preserved
as-is in the doc for the record, in case the user's situation ever
genuinely changes, but nothing in this pass moves the codebase one inch
closer to it.

What the doc's §5 ("Schema-Growth Concerns") actually called for, though,
is real GoDaddy-MySQL work that helps at *any* scale this project will
ever run at, not just a hypothetical 100k-student future — so that part
was implemented and tested for real:

- **Generalized ledger-content archival**, extending the one-off pattern
  §5 references (ai_messages: "keep the summary row, drop the body") to
  every other high-volume ledger with a genuinely large per-row body:
  `code_executions` (`stdin`/`stdout`/`stderr`) and `exam_responses`
  (`response`/`grader_feedback`). `cron/archive-ledger-content.php`,
  driven by `config/archival_policies.php` (the same config-as-code
  pattern as `ai_tiers.php`/`cadences.php`), exports the full row as one
  JSON line to local disk (`storage/app/archives/{table}/{YYYY-MM}.jsonl`,
  grouped by the row's *real* age — `exam_responses` has no honest
  timestamp of its own, so its age is read from the parent
  `exam_attempts.submitted_at` via a join, not row-insertion time) before
  stripping the body in place. Verified end to end: old rows (15/8/25
  months, respectively) were correctly archived-then-stripped while
  genuinely recent rows in the same tables were left completely untouched,
  and the archive files' contents matched the original pre-strip data
  exactly. A small real gap-fill was needed first — `exam_responses` had
  no `created_at` column at all to determine row age from; added going
  forward, with the archival cron using the parent attempt's
  `submitted_at` rather than relying on it for already-existing rows.
  `credit_transactions`/`xp_transactions`/`communication_logs` are
  deliberately excluded from this cron — their rows are already small
  (no large body column), so there's nothing to strip.
- **A real, tested bug caught while building this**: none this round in
  the archival cron itself, but the **MySQL partitioning piece below**
  surfaced a genuine, previously-unknown schema constraint.
- **MySQL native partitioning — attempted for real, found to be
  impossible on this schema as designed, documented honestly rather than
  shipped broken.** §5 calls partitioning "the mechanical complement to
  archival." Writing the actual `ALTER TABLE ... PARTITION BY RANGE`
  migration and running it against a real MySQL 8.4 instance surfaced:
  `ERROR 1506: Foreign keys are not yet supported in conjunction with
  partitioning` — an InnoDB restriction that blocks partitioning any table
  with a foreign key in *either* direction. Every single one of the five
  candidate tables has at least one outbound FK
  (`credit_transactions`→`credit_wallets`/`payments`,
  `xp_transactions`/`communication_logs`→`users`,
  `code_executions`→`code_workspaces`/`users`,
  `ai_messages`→`ai_conversations`) — none of them can be partitioned
  without first dropping that FK and re-enforcing referential integrity in
  application code instead, a materially bigger tradeoff than "run a
  migration" (it removes a DB-level guarantee, e.g. `ON DELETE CASCADE`,
  that would need a PHP equivalent). `database/partition-ledger-tables.sql`
  was rewritten from a runnable migration into an honest reference
  document explaining exactly this, once the first `ALTER TABLE` in it
  failed against a real database — rather than leaving a script that looks
  runnable but silently doesn't work. The actual, sufficient mitigation at
  any GoDaddy-only scale this project commits to: archival (above) bounds
  the tables that have a body worth bounding; the lean tables already
  query through indexed columns (`wallet_id+created_at`,
  `student_id+created_at`, `user_id+trigger_event`) regardless of total
  row count, so they stay query-fast even unpartitioned, just not
  space-bounded the way a rotated partition would be.
- **`domain_events` pruning** — §5's one explicit exception to "never
  delete a ledger row": once both `processed_by_admin` and
  `processed_by_student_app` are true and a retention window (30 days) has
  passed, a row has done its entire job and has no ongoing audit value.
  `cron/prune-domain-events.php` does exactly that deletion, nothing more.
  Verified against three seeded rows (old+fully-processed,
  recent+fully-processed, old+still-unprocessed) — only the first was
  pruned; the other two were correctly left alone.

## Phase 8 — Infrastructure & DevOps (final phase)

`docs/student-module/08-infrastructure-devops.md` — the last of the 35
documents. Same pattern as Phase 7: the doc covers both an AWS path and a
GoDaddy path, and only the GoDaddy-compatible, no-cloud-violating pieces
were actually implemented. ECS/ECR/Aurora/CloudWatch/Terraform/S3 content
is reference material, correctly transcribed, intentionally not built —
consistent with the standing 3-year mandate every phase in this program
has honored.

**Two real, previously-missing monitoring jobs, built from a doc
requirement that turned out to have zero existing implementation**:

- **`cron/reconcile-credit-ledger.php`** — §3: "any nonzero drift pages
  someone immediately... drift specifically means an application bug
  exists." `credit_wallets.credits_balance` is denormalized by its own
  column comment ("maintained by trigger/app — never computed ad hoc"),
  but nothing anywhere in this codebase had ever actually checked it
  against the `credit_transactions` ledger it's supposed to mirror.
  "Pages someone" means a real email (via `App\Core\Notifier`) to every
  active SuperAdmin — there's no PagerDuty in a no-cloud build, but the
  underlying signal still needs a human, not a log line nobody reads.
  Verified end to end with a deliberately-seeded drifted wallet (stored
  50, ledger summed to 80): correctly flagged, correctly alerted via both
  in-app and a real captured email with the exact expected/actual numbers,
  while a second, non-drifted wallet correctly triggered nothing.
- **`cron/check-support-sla-breaches.php`** — §3: SLA breaches were
  "already specified at the application level as 'alerts a supervisor'"
  but nothing computed one. A breach: no first response yet, and the
  ticket's category `sla_hours` window (already in the schema, never
  read against ticket age before this) has passed. Verified with one
  ticket past its 4-hour SLA window (correctly alerted, exact hours-open
  figure in the message) and one still within it (correctly silent).

**`HealthController` gained a `metrics()` endpoint** — §3's GoDaddy-specific
constraint: no CloudWatch-equivalent exists under shared hosting, so
monitoring "leans more heavily on application-level `/health` endpoints
polled by an external uptime service." `/health` stays a fast liveness
ping; the new `/health/metrics` exposes communication-queue backlog,
today's AI spend, and the last reconciliation/SLA-check results — real
numbers from real tables, verified by polling it after deliberately
triggering both checks above. `ai_gateway_retries_today` is the honest
substitute for §3's literal "AI Gateway fallback-to-secondary-provider
rate" — **there is no secondary provider in this build, by design** (one
accepted AI dependency, no failover route), so that exact metric doesn't
apply; the real equivalent leading indicator (how often the one route
needed its single transient-failure retry) is counted from today's log
file, since `AiGateway` doesn't persist a queryable retry flag anywhere —
stated as a real, current limitation, not silently faked with a made-up number.

**`cron/backup-database.php`** — §4 (GoDaddy path) calls for "an
independent nightly mysqldump piped directly to S3." S3 is out per the
no-cloud mandate, so this writes local, gzip-compressed, rotated (14-day)
backups instead — and the README states plainly what that does and
doesn't provide: it protects against cPanel's own backup product failing
or being misconfigured, but **not** against the account itself being
compromised or suspended, since the backup lives on the same account as
the live data. A true independent-location backup needs a second
location by definition; there's no way to honestly provide one within a
zero-external-services commitment. Verified for real: a genuine
`mysqldump | gzip` run produced a valid, complete 132-table archive with
real seeded data confirmed inside it, the failure path was exercised
(missing `gzip` on the Windows test machine correctly produced a logged
critical error and no zero-byte file), and rotation correctly pruned a
20-day-old fake backup while keeping the current one.

**`scripts/check-additive-schema-changes.php`** — §1's migration
discipline: "every migration touching a table the Admin app also
reads/writes is additive-only... and ships with a check." This codebase
has no migration framework, so the check is a standalone script scanning
`schema_student_portal.sql` for DROP/RENAME/MODIFY/CHANGE statements and
cross-referencing every touched table against the Admin codebase. Running
it for real against this build's own accumulated schema file surfaced 4
pre-existing `MODIFY COLUMN` statements from earlier phases (all
backward-compatible ENUM-value widenings on `exams.type`,
`live_classes.status`, `attendance.status`, and
`assignment_submissions.status` — manually reviewed just now and
confirmed safe, none narrow or remove an existing value). These predate
this script and aren't being undone; the script's job is catching the
*next* one before it ships, which is exactly what "ships with a check"
means in practice. It also correctly found 6 tables `schema_student_portal.sql`
touches that the Admin codebase also references directly — printed as a
review reminder, not a hard failure, since a grep hit doesn't itself
prove incompatibility.

**Reference-only, not fully testable in this environment** (no real git
remote or GoDaddy SSH target exists to deploy to): `.github/workflows/deploy-student-portal.yml`
documents the CI/CD shape §1 calls for, adapted to this actual codebase —
no `composer install` (zero runtime Composer dependencies by design) and
no `artisan migrate`/cache-rebuild step (no migration framework or cache
layer exists here), with the schema-discipline script wired in as a real
gate before deploy. `database/seed-staging-fixtures.sql` is a synthetic,
no-real-PII fixture set for §2's "staging never holds real student PII"
rule — verified to actually import cleanly against a real schema,
including realistic parent-student-consent relationships.

§§2 (environment topology) and most of §4 (AWS-specific DR) are reference
material only, beyond what's covered above.

## Local development

```
cd api
cp .env.example .env   # fill in DB_* for your local MySQL
php -S localhost:8001 index.php
```

For the frontend, any static file server works (e.g. `npx serve web`), or
just open `web/index.html` directly — `assets/js/api.js` calls relative
`/api/v1/...` paths, so point a reverse proxy (or your local server) at the
same origin/path as the API so that assumption holds, the same way it will
in production.
